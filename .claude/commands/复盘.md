# /复盘 — 对话价值榨取

**你必须执行这个工作流，不是描述它。**
**严格遵守阶段顺序，每完成一个阶段输出完成标记。**
**每个阶段必须完整执行。即使该命令在同一对话中已被调用过、或本次对话与上次复盘之间新增内容较少，也不允许依赖记忆或前次执行结果省略任何步骤。**

## 核心目标

扫描整个对话，识别所有"值得注意的时刻"，进行领域化深度分析，对照已有知识资产，将发现沉淀为 memory 文件。

## 执行流程

### [前置] 加载历史背景（自动，不打扰用户）

1. 读取 `C:\Users\啦啦啦啦啦\.claude\projects\D--\memory\MEMORY.md`
2. 提取当前对话中的关键主题词：扫描对话中出现频率最高的 5-10 个领域术语和技术名词（如 "PPG""MAE""decimate""memory""skill"），作为 Grep 搜索词
3. 用 Grep 在 memory 目录中搜索这些主题词
4. 如果找到相关 memory 文件（≤5条），静默加载作为后续追问的参考背景
5. 如果 MEMORY.md 不可用，跳过，标注"无历史背景"

**输出**：列出具体加载的 memory 文件名（而非仅数量）：
`✅ [前置] 历史背景加载完成（N 条: <文件名1>, <文件名2>, ...）`

### 阶段1: 思考 — 扫描对话

**扫描整个对话**（不允许只扫描"上次复盘之后的新增内容"——每次 /复盘 都是对完整对话的独立复盘），找出以下六类"值得注意的时刻"：

| 类型 | 定义 | 信号 |
|------|------|------|
| **错误/失败** | 搞错了什么，不得不回头修 | Agent 重做了某事、用户纠正了 Agent、报错后改了方案 |
| **意外** | 结果跟预期不同（无论好坏） | "居然""没想到""出乎意料"、结果大幅偏离预期 |
| **决策** | 做了一个有取舍的选择 | 在两个方案间选了其一、明确放弃了一条路径 |
| **发现** | 学到了新东西 | "原来""发现了""注意到了一个模式" |
| **恢复** | 从失败中怎么爬出来的 | 排错过程、根因分析路径、修复步骤 |
| **效率** | 做得特别顺或特别慢的地方 | 某段交互特别高效/某件事反复折腾 |

**判断领域**（标签匹配，一个时刻可以属于多个领域）：
- experiment: ML实验、模型训练、超参调优、消融研究
- data: 数据下载、预处理、格式转换、完整性验证
- coding: 写代码、修bug、代码审查
- research: 文献搜索、技术调研、方法对比
- debugging: 排错、根因分析、异常排查
- design: 系统架构设计、技术选型、流程设计、设计审查
- 其它：根据对话内容自行判断，不确定时标注"待确认"

**跳过判断**：如果对话属于纯日常闲聊、简单信息查询，或内容类型超出当前 skill 体系覆盖范围且无法为此新建领域 → 明确告知"此对话不适合复盘"，说明原因，终止。

**阶段1 必须输出**（不允许跳过或概括）：
- 扫描到的每个时刻，按类型分组列出，含简要说明和归属领域：
  ```
  [错误/失败]:
  - <简要描述> → 领域: <domain>
  [意外]:
  - ...
  [决策]:
  - ...
  [发现]:
  - ...
  [恢复]:
  - ...
  [效率]:
  - ...
  ```
- 涉及领域列表及对应时刻数量
- 如无任何时刻 → 输出跳过判断原因并终止

### 阶段2: 提取 — 领域化深度分析

**对每个领域**，按以下步骤进行：

#### 有 retro-<domain> skill 的情况
1. 读取对应的 skill 文件全文（在 `~/.claude/skills/retro-<domain>/SKILL.md` 或项目 `.claude/skills/` 下）
2. 按 skill 中的评估维度、反模式/正向模式清单、关键追问，对该领域的每个时刻进行深度分析
3. 如果本次发现的坑未被 skill 的反模式清单覆盖 → 直接追加到 skill 文件的"常见反模式"表格，新增一行（标注 `🤖 自动添加 YYYY-MM-DD，待确认`）
4. 如果本次发现了一个值得固化的好做法（符合 skill 正向追问的检测信号）未被 skill 的"已固化模式"表格覆盖 → 直接追加到该表格，新增一行（标注 `🤖 自动添加 YYYY-MM-DD，待确认`）

#### 没有 retro-<domain> skill 的情况（正常路径，不是降级）

执行以下流程创建 skill 并复盘。该流程会暂停等待用户确认。

1. **搜索行内经验**：用 WebSearch 搜索该领域的公认评估标准、常见反模式、最佳实践（如 "<domain> retrospective best practices common pitfalls"）
2. **结构化整理**：将搜索结果组织为：评估维度（3-7个）、常见反模式（3-5个带检测信号）、关键追问（3-5个）
3. **对抗性审查**：审查生成的 skill 草案——是否存在明显盲区、系统性偏向，以及是否覆盖了本次对话涉及的核心问题
4. **展示给用户确认**：呈现整理好的 skill 草案，问"保存为 retro-<domain> skill 吗？"
5. 用户确认后：保存到 `~/.claude/skills/retro-<domain>/SKILL.md`
6. **用新 skill 复盘当前对话**

**搜索失败时的处理**：如果步骤 1 的 WebSearch 超时、返回空或低质量结果，触发降级——降级为 AAR 4 问通用框架处理该领域，标注"搜索无结果，skill 未生成"，继续后续流程。其他异常场景见底部降级表及通用降级规则。

**迭代机制**：深度分析过程中如发现阶段1遗漏的新"时刻" → 追加到发现列表 → 对该新时刻归类+领域判定 → 进入阶段2分析（不重新扫描整个对话，只处理新增项）。

**多领域归属**：一个发现涉及多个领域 → 归属到最相关领域深度复盘，其他领域在综合时简述。

**阶段2 必须输出**（每个领域不允许省略）：
```
[retro-<domain>]:
- skill 加载: ✅/⚠️(降级原因)
- 关键发现:
  1. <具体发现> — <对应评估维度>
  2. ...
- 反模式匹配: <命中 X 条>/<新增 Y 条>
- 正向模式匹配: <命中 X 条>/<新增 Y 条>
```

### 阶段2.5: 对照 — 存量知识更新

**对每个发现**，与已有知识资产对照：

1. **匹配已有资产**：读取 MEMORY.md → 按标题+主题词筛选候选 → 读取具体文件
2. **判定关系**（以下"条目"泛指已有的 lesson/insight/rule/pattern）：
   - 同主题+结论一致+新信息仅是表述方式不同或补充细节 → **验证**（更新验证计数+追记证据）
   - 同主题+结论一致+新信息提供不同视角/重要补充 → **补充**（追加内容到已有文件）
   - 已有条目部分错误/过时 → **修正**（保留原文用~~删除线~~ + 追加修正 + 标注原因和日期）⚠️需确认
   - task级条目（含 memory 和 skill 中的已固化模式）+本次是第3+次独立验证（或用户手动触发） → **升级**。教训升级为 project/user 级 rule；已固化模式升级为 CLAUDE.md 规则或独立 skill
   - 两个已有文件讲同一件事 → **合并**（合并后删冗余）⚠️需确认
   - 新旧结论冲突+无法简单判断 → **矛盾**（呈现双方论据，用户裁决）⚠️需确认
   - 无匹配 → **新建**

3. **执行规则**：
   - 纯新增、验证计数+1、简单追加 → 自动
   - 修正已有内容、合并文件、写 CLAUDE.md、新建/修改 skill、矛盾 → ⚠️**需用户确认后执行**

完成标记：`✅ 阶段2.5: 对照完成（验证 N 条, 补充 M 条, 新建 K 条, 需确认 X 条）`

---

### 阶段3: 沉淀 — 写入

按发现性质路由：
- **Rule（规则）**：违反后有明确负面后果 → CLAUDE.md 或 skill 指令 ⚠️需确认
- **Lesson（教训）**：从错误/失败中学到的，含反证条件 → `memory/feedback_<slug>.md`
- **Insight（洞察）**：深层因果理解，可被后续引用 → `memory/reference_<slug>.md`
- **Pattern（模式）**：可复用的做法/模板 → skill 的"已固化模式"表（正向模式，含 verified_count）。已验证 3+ 次的模式 → 升级为 CLAUDE.md 规则或独立 skill ⚠️需确认
- **Action（行动）**：需后续跟进 → `memory/action_<slug>.md`

每个 memory 文件格式：
```markdown
---
name: <kebab-case-slug>
description: <一句话摘要>
metadata:
  type: feedback | reference | action
  domain: <领域标签>
  verified_count: <N>
  last_verified: <YYYY-MM-DD>
---

<具体内容>

**Why:** <为什么重要>
**How to apply:** <如何使用>
**反证条件:** <什么情况下这个结论是错的？>
```

**写入 memory 后**，更新 `C:\Users\啦啦啦啦啦\.claude\projects\D--\memory\MEMORY.md`，在末尾追加一行索引。

**Skill 反模式补充**：直接追加到对应 skill 文件的"常见反模式"表格（标注 `🤖 自动添加 YYYY-MM-DD，待确认`）。

完成标记：`✅ 阶段3: 沉淀完成（写入 N 个文件）`

## 降级路径

降级路径处理**异常/失败**场景。以下为预设的常见异常及处理方式。遇到预设表未覆盖的异常时，遵循底部"通用降级规则"。

遇到任何步骤失败，**不得隐藏错误信息，不得中断整体流程**。标注失败原因，跳过该步骤，继续后续流程。

| 失败场景 | 处理 |
|----------|------|
| MEMORY.md 不存在/格式异常 | 跳过[前置]和对照，标注"无历史资产" |
| 对话过长（>200轮） | 只扫描最近 50 轮 + 开头 10 轮 |
| WebSearch 返回空/低质量 | 降级为 AAR 4 问通用框架，标注"搜索无结果，skill 未生成" |
| WebSearch 超时/网络错误 | 重试 1 次，仍失败→降级为 AAR 4 问 |
| Skill 文件存在但格式损坏 | 跳过该 skill，降级为 AAR 4 问处理该领域 |
| 迭代发现新时刻 >10 次 | 截断，标注"建议人工审查" |
| Memory 文件写入失败 | 在对话中输出完整内容，提示用户手动保存 |
| MEMORY.md 更新失败 | 在对话中输出应添加的索引行 |

**通用降级规则**：遇到预设表未覆盖的新异常时——先判断错误类型（临时/永久/致命）。临时错误重试 1 次。尝试搜索解决方案。无法解决则跳过该步骤，标注 `⚠️ 步骤X异常: <描述>`，继续后续流程。复盘结束时在汇报中说明所有跳过的步骤及原因。

## 注意事项

- 阶段1 的完整性决定了整个复盘质量。仔细扫描，不要遗漏。
- **平衡锚定**：复盘的目的不仅是发现问题。同等重视两方面——值得固化复用的好做法（来源：六类时刻中的"发现"和"效率"），以及需要改进的问题（来源："错误/失败""恢复"）。
- 每个发现必须包含"反证条件"——什么情况下结论是错的？防止确认偏误。
- 写 memory 时复用已有命名约定（feedback_/reference_/action_ 前缀）。
- 如果用户提供了 $ARGUMENTS，聚焦于该主题。如果未提供，复盘整个对话。
- 复盘结束后，在对话中向用户呈现阶段4汇报，并指出需要用户确认的事项。
