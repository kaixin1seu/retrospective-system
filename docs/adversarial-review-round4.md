# 对抗性审查 第四轮：可行性重设计 + 行为陷阱缓解 + 残存盲区

> 基于第三轮发现的约束（单Agent串行、检索限制、行为陷阱）
> 本轮目标：在约束下重设计，直到零致命问题

---

## 挑刺方 🔴

### 聚焦1: 单Agent串行约束下的诚实重设计

第三轮揭穿了一个关键假设：**"并行调用多个 lens"在单 Agent 对话中不可实现**（除非用 Workflow，但 /retro 是对话中的命令）。

这意味着 v2 的架构图有误导性。实际可行的架构是：

**现实版架构**：
```
/retro [quick|standard|deep]
→ Agent 收到 retro 框架 prompt
→ Agent 判断涉及的领域（1-3个）
→ Agent 串行 Read 相关 retro-* skill 文件（每个约 1-2K tokens）
→ Agent 融合所有 lens 的评估维度，一次性执行复盘
→ Agent 产出结构化结果 + 写 memory 文件
```

**挑刺**：这个串行流程中，"并行"的优势完全丧失了。但换个角度看——在单 Agent 对话中，Agent 本来就是在串行思考。融合多个 lens 到一个思考过程中，可能反而比"假装并行"更有效。这个约束不是 bug，是现实。

→ **裁决**：接受串行约束。将"并行调用"改为"融合多 lens 视角的一次性深度复盘"。这不是降级，是更诚实的实现。

### 聚焦2: Pre-Task 检索的可行实现

第三轮指出：语义向量搜索超出 Claude Code 原生能力。

**可行的替代方案**：
1. **MEMORY.md 索引**（已有）：自动加载到每个会话，Agent 可看到所有 memory 的标题和摘要
2. **CLAUDE.md 检索指令**：在 CLAUDE.md 中加入："在开始新任务前，检查 MEMORY.md 中是否有相关的 lessons learned"
3. **标签/关键词索引**：在 memory 文件的 frontmatter 中加入 `tags` 字段，Agent 通过 Grep 搜索标签
4. **手动检索**：用户在执行重要任务前，主动运行 `/retro-search <关键词>`

**挑刺**：方案 1+2 已经存在！用户的 MEMORY.md 在每次会话自动加载，Agent 已经能看到所有 memory。问题是 Agent **是否主动去检索和应用**。增加 CLAUDE.md 中的"检索指令"可以让这个行为更可靠。

→ **裁决**：不需要额外的检索系统。在 CLAUDE.md 中加入"任务开始前检索相关 memory"的指令 + 在 memory 文件的 frontmatter 中规范 tags 字段 = 80% 的检索效果，20% 的实现成本。

### 聚焦3: 行为心理学陷阱的缓解设计

**3.1 复盘疲劳 → 缓解**
- `/retro` 不主动提示，完全手动触发
- 快速模式目标：用户只需确认 3 个问题，系统自动完成剩下的
- 设计为"仪式"而非"任务"：复盘是收尾的满足感，不是额外的负担

**3.2 确认偏误放大 → 缓解**
- 每个复盘发现必须包含"反证条件"：什么情况下这个结论是错的？
- 定期（每 5 次复盘）运行一次"反模式审查"——刻意寻找与已有结论矛盾的证据

**3.3 规则自我强化 → 缓解**
- 规则升级需要"独立验证"：同一个规则必须在至少 2 个不同的任务中被独立触发，才能升级作用域
- 单次任务中重复出现的同一问题只计为 1 次

**3.4 "复盘即完成"错觉 → 缓解**
- 复盘产出 action items 必须进入 task list，下次任务开始前检查
- 在 memory 文件的 frontmatter 中加入 `applied_count` 字段，追踪规则被实际应用的次数
- 超过 30 天未被应用的规则自动标记为 `stale`

### 聚焦4: 之前轮次可能遗漏的盲区

**4.1 与用户现有 skill 生态的关系**
用户有 100+ skills，其中包含一些与复盘直接相关的：
- `retro`（已存在！需要检查是否冲突）
- `code-review`（代码审查）
- `post-mortem`（事后分析）
- `pre-mortem`（事前分析）
- `verification-before-completion`（完成前验证）
- `skill-review`（skill 审查）

**挑刺**：`retro` skill 已经存在了！我们的 `/retro` 命令会和它冲突吗？需要先检查现有 `retro` skill 的内容。

**4.2 复盘系统的"元"问题**
- 如果复盘系统本身出了 bug，用户怎么知道？
- 如果复盘持续产出低质量教训，用户怎么发现？
- 系统需要自诊断能力：定期检查"最近 5 次复盘产出的教训中，有多少被实际应用了？"

**4.3 用户当前的实验节奏**
用户的 ML 实验节奏非常快（从 memory 看，v5→v5b→v5c 在短时间内完成）。在这么快的节奏下：
- 如果每次实验后都做标准复盘，会不会打断节奏？
- 也许应该设计"实验系列复盘"——在一组相关实验（如 v5→v5b→v5c）完成后，做一次综合复盘
- 这比单个实验复盘更有价值，因为能看到演进路径

**4.4 复盘产物的"可检索性"退化**
- 随着 memory 文件增长（当前 34 个 → 未来可能 100+），MEMORY.md 索引本身会变得很长
- Agent 在上下文窗口中能看到整个 MEMORY.md，但注意力会随索引长度下降
- 需要设计索引的分层/折叠机制

### 聚焦5: 设计文档本身的问题

**5.1 使用示例过于理想化**
v2 设计文档中的使用示例展示了一次"完美"的复盘。但实际情况可能是：
- 用户累了一天不想复盘
- 任务很无聊没什么可复盘的
- 复盘产出的"教训"其实是 trivial 的
- 系统需要能识别"这次任务没什么特别的"并快速结束

**5.2 没有考虑"不需要复盘"的情况**
不是每次任务都值得复盘。如何区分"值得复盘的任务"和"日常琐事"？
- 任务持续时间（< 5 分钟的任务可能不值得）
- 是否有意外结果（错误、惊喜、偏离预期）
- 用户是否主动表示学到了什么

## 仲裁裁决 ⚖️

### 🔴 致命 — 本轮发现

**1. 已有 `retro` skill 冲突**
→ 必须先检查现有 skill，避免命名冲突。可能需要改名或整合。

**2. "不需要复盘"的判断缺失**
→ 不是每次任务都值得复盘。需要垃圾过滤机制。

### 🟡 重要 — 需要设计调整

3. 行为陷阱缓解需要融入设计（反证条件、独立验证、applied_count 追踪）
4. Pre-Task 检索方案过度设计了——利用已有 MEMORY.md 加载 + CLAUDE.md 检索指令即可
5. 设计文档中的使用示例过于理想化，需要加入"不完美场景"

### 🟢 优化
6. 与现有 skill 生态的深度整合（后续）
7. 复盘系统的自诊断（后续）
8. 实验系列复盘模式（后续）

### ⚪ 无需修改
- 串行约束下的架构调整：这已经在 v2 设计的能力范围内，不需要架构变动，只需文档措辞从"并行"改为"融合多lens视角"

致命问题从第三轮的 3 个降到本轮的 2 个（且都是可快速解决的：检查命名冲突、增加垃圾过滤）。
