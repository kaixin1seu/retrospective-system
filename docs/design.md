## 0. 核心设计

### 要解决的问题

在一个对话中完成重要任务后，输入 `/复盘`，榨干整个对话的价值——既包括问题（错误、教训、决策失误），也包括好做法（可复用模式、有效方法论）。所有发现沉淀为知识资产，**同时维护存量知识**——更新、修正、合并已有 memory/skill/rule。

### 第一性原理

复盘的本质是**维护一个不断进化的知识库**。每次复盘做两件事：
1. **入库**：从经验中提取新知识（问题 + 好做法）
2. **盘点**：对照已有知识，更新、修正、合并、升级

核心洞察：**复盘的结构是通用的（AAR 4问、元认知5阶段），但评估标准因领域而异**（"好代码"vs"好实验"的判断标准不同）。

→ 方案：**通用框架（`/复盘` 命令）+ 领域透镜（`retro-<domain>` skill）**

## 1. 完整流程

```
/复盘
    │
    ├─ [前置] 跨 session 上下文（自动，不打扰用户）
    │   Agent 静默扫描 MEMORY.md → 提取高频技术术语作为 Grep 搜索词
    │   → 加载 ≤5 条最相关 memory 作为追问参考背景
    │
    ├─ 阶段1: 思考 — 扫描对话
    │   六类时刻：错误/失败、意外、决策、发现、恢复、效率
    │   七个领域：experiment, data, coding, research, debugging, design, 其它
    │   跳过判断：纯闲聊/简单查询/超出 skill 覆盖且无法新建 → 终止
    │
    ├─ 阶段2: 提取 — 领域化深度分析
    │   │  对每个领域：
    │   │  ├─ 有 skill → 读全文 → 按评估维度 + 反模式/正向模式 + 追问 → 深度分析
    │   │  │   ├─ 新坑未被反模式覆盖 → 追加到"常见反模式"表
    │   │  │   └─ 好做法未被正向模式覆盖 → 追加到"已固化模式"表
    │   │  └─ 没有 skill → 搜索行内经验 → 结构化 → 审查 → 用户确认 → 保存 → 复盘
    │   │       （搜索失败时才降级到 AAR 4 问通用框架）
    │   │
    │   │  迭代：分析中发现新时刻 → 追加列表 → 仅对新项归类+领域判定 → 继续分析
    │   │  多领域：归属最相关领域深度复盘，其他领域综合时简述
    │   │
    │   ├─ 阶段2.5: 对照 — 每个发现 vs 已有知识资产
    │   │   匹配已有 memory/skill/rule → 判定关系：
    │   │   - 验证（同主题+一致+新信息是细节补充）→ 自动：计数+1
    │   │   - 补充（同主题+一致+新信息是重要新视角）→ 自动：追加
    │   │   - 修正（已有条目部分错误/过时）→ ⚠️确认（保留原文+追加修正）
    │   │   - 升级（task级+第3+次独立验证）→ 自动：升级作用域
    │   │   - 合并（两个文件同一件事）→ ⚠️确认
    │   │   - 矛盾（新旧冲突无法判断）→ ⚠️确认（用户裁决）
    │   │   - 新建（无匹配）→ 自动
    │   │
    │   ├─ 阶段3: 沉淀 — 按性质路由
    │   │   Rule(规则)    → CLAUDE.md 或 skill 指令    ⚠️确认
    │   │   Lesson(教训)  → memory/feedback_*.md         自动
    │   │   Insight(洞察) → memory/reference_*.md        自动
    │   │   Pattern(模式) → skill 的"已固化模式"表       自动（表格追加）
    │   │                   或独立 command                ⚠️确认（新文件）
    │   │   Action(行动)  → memory/action_*.md           自动
    │   │
    │   └─ 更新 MEMORY.md 索引
    │
    ├─ 阶段4: 汇报
    │   🟢 值得固化的（好做法/正向模式）：
    │     1. <做法> — <为什么有效>
    │   🔴 需要改进的（问题/教训）：
    │     1. <教训> — <根因+影响>
    │   📋 统计 + ⚠️需确认 + 💡skill更新 + 📌闭环引用率
    │
    └─ [闭环验证]
        检查上次产物在本对话是否被引用。连续 5 次未引用 → 标记审查。
```

## 2. 核心机制

### 2.1 平衡锚定

复盘不是只找问题。同等重视两方面：

| 方向 | 识别机制 | 沉淀位置 |
|------|----------|----------|
| 🔴 问题/教训 | skill 反模式表 + 挑刺追问 + 检测信号 | memory feedback_* / skill 反模式表 |
| 🟢 好做法/模式 | skill 已固化模式表 + 正向追问 + 检测信号 | memory reference_* / skill 已固化模式表 |

每个 skill 的"正向追问"附带检测信号（类似反模式的检测信号），让 Agent 像识别反模式一样精确地识别值得固化的好做法。

### 2.2 Skill 自维护

复盘不仅产出 memory，还维护 skill 本身：
- 发现新反模式 → 追加到 skill 的"常见反模式"表
- 发现新正向模式 → 追加到 skill 的"已固化模式"表（含 `verified_count` 列）
- 追加时标注 `🤖 自动添加 YYYY-MM-DD，待确认`

### 2.3 已固化模式生命周期（读路径闭环）

正向模式不是 write-only。三阶段升级到日常行为：

| 阶段 | 条件 | 位置 | Agent 如何读到 |
|------|------|------|---------------|
| 孵化 | 1 次复盘发现 | skill 已固化模式表（verified=1） | 仅复盘时加载 |
| 验证 | 3+ 次独立验证 | 同上（verified≥3）→ 阶段4 汇报提示可升级 | 复盘时看到升级建议 |
| 生效 | 用户确认升级 | CLAUDE.md 规则或独立 skill | **每次会话自动加载**或按需调用 |

升级路由：行为规则 → CLAUDE.md；可复用流程 → 独立 skill；领域知识 → memory scope 提升。

### 2.4 存量知识更新（7 种关系）

### 2.4 存量知识更新（7 种关系）

每次复盘的发现都与已有资产对照。语言上中性化——"条目"而非"教训"——因为发现可能是正向模式。

### 2.5 降级路径

基于 AI Agent 多步任务可靠性研究（每步 99% → 20 步后仅 ~82%），为关键步骤预设降级策略。三类错误：Transient（重试1次）、Permanent（跳过+标注）、Critical（保存产出+通知）。

12 种预设降级场景 + 通用降级规则（未知异常 → 分类 → 搜索方案 → 跳过 → 标注）。

### 2.6 闭环验证

每次复盘检查上次产出在本次对话中是否被引用。连续 5 次未引用 → 标记审查。解决"复盘产物是否被使用"的验证问题。

## 6. 设计原则

1. **质量优先**：skill 一次打磨，反复受益。复利。不因效率牺牲质量。
2. **平衡锚定**：同等重视问题识别和正向模式固化。复盘不是只找错。
3. **行内经验优先**：不从对话临时归纳，学习行内人踩过的坑和验证过的做法。
4. **入库 + 盘点并重**：不只写新文件，更维护存量——验证、补充、修正、合并、升级。
5. **闭环验证**：不止产出，更要追踪是否被后续 Agent 引用使用。
6. **对抗性审查内置**：新 skill、新规则、修正写入前必审查；每次 `/复盘` 的新领域创建流程自带审查步骤。
7. **矛盾不自动解决**：新旧冲突交给用户裁决。Agent 提供论据，用户做决定。
8. **失败不静默**：任何步骤失败 → 标注跳过 → 继续后续流程 → 汇报时说明。
9. **渐进增长**：领域 skill 随使用积累，反模式和正向模式表随复盘自填充。
